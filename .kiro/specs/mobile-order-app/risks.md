# Risks

1. **サーバ負荷**
   - ピーク時の同時注文で在庫引当が競合する可能性
   - 対策: トランザクション制御、capacity_used の原子的更新

2. **UXリスク**
   - スロット満杯や売切れ時のエラーメッセージが不十分だと顧客満足度低下
   - 対策: UIで明示的に「満杯」「売切れ」を表示

3. **セキュリティ**
   - 注文ID推測による不正アクセス
   - 対策: UUID利用、APIキーで店側認証

4. **開発スケジュール**
   - フェーズごとの遅延が全体に波及
   - 対策: MVPに絞り込み、優先度の高い機能から実装

5. **運用リスク**
   - 店舗スタッフが管理画面に慣れず運用負担が増加
   - 対策: 管理UIをシンプル化し、マニュアル整備
